# Day 10 Reliability Final Report

## 1. Architecture summary

The Reliability Gateway coordinates request routing using a layered mechanism designed to maximize availability, minimize cost/latency, and protect downstream dependencies from overload.

```
                    User Request
                         |
                         v
           [Reliability Gateway Routing]
                         |
            +------------+------------+
            |                         |
     [Cache Check]                    | Cache Miss
            |                         v
            | HIT             [Provider Chain]
            v                         |
      Return Cached        +----------+----------+
                           |                     |
                   [Primary Provider]   [Backup Provider]
                           |                     |
                    Circuit Breaker       Circuit Breaker
                           |                     |
                           +----------+----------+
                                      |
                                      | All Providers Fail
                                      v
                           [Static Fallback Response]
```

### Components:
- **Reliability Gateway**: The core entry point that orchestrates routing across the Cache, Circuit Breakers, and LLM Providers.
- **Semantic Cache**: Resolves incoming queries using Cosine Similarity of character 3-grams and word tokens. It prevents duplicate upstream calls, saves cost, and enforces privacy (via regex filters) and false-hit detection (year/number mismatch).
- **Circuit Breaker**: Implements a 3-state machine (`CLOSED`, `OPEN`, `HALF_OPEN`) per provider to fail fast and prevent retry storms.
- **Provider Chain**: Order-based fallback chain (`primary` -> `backup`) that tries next providers if previous ones fail or their breakers are open.
- **Static Fallback**: A last-resort degraded message returned when all providers fail or are tripped.

## 6. Redis shared cache

Shared caching is crucial for production because:
- **In-memory cache is local**: If a gateway is scaled horizontally across multiple instances or running in stateless Docker containers, each instance's cache is isolated. This results in duplicate upstream API calls and wasted costs.
- **SharedRedisCache solves this**: By externalizing cache storage to a centralized Redis cluster, multiple gateway instances share a unified, persistent cache state. Any cached response populated by instance A is immediately available to instance B.

### Evidence of shared state

We verified shared cache state using two separate cache instances accessing the same Redis keys:
```
c1 = SharedRedisCache(redis_url="redis://localhost:6379/0", prefix="rl:test:shared:")
c2 = SharedRedisCache(redis_url="redis://localhost:6379/0", prefix="rl:test:shared:")
c1.set("shared query", "shared response")
# Instance c2 successfully gets the key set by instance c1:
c2.get("shared query") -> ("shared response", 1.0)
```

## 8. Failure analysis

### Remaining Weaknesses:
1. **Local Circuit Breaker State**: Currently, circuit breaker state is stored in memory. In a multi-instance gateway, if Provider A goes down, each instance has to fail 3 times before opening its local breaker, causing redundant failure calls to the downstream provider.
2. **Single Point of Failure (Cache)**: If Redis goes down, the cache call fails. We should catch Redis exceptions and gracefully degrade to in-memory caching or bypass caching entirely (graceful degradation).

---

## 9. Next steps

1. **Redis Circuit Breaker State**: Store circuit state, failure counts, and state transitions in Redis using `INCR` and `EXPIRE` so all instances fail-fast collectively.
2. **Redis Cache Fallback**: Add try-except around Redis calls; if Redis is down, fall back to a local in-memory cache (`ResponseCache`) to prevent gateway request failures.
3. **Budget/Cost Cap Routing**: Track estimated daily/monthly cost and dynamically disable expensive providers once a cost threshold is reached, defaulting to cache or cheaper fallback models.
